# Neo4j load: the FULL Track B translational-evidence graph

These files let you load the **entire** un-capped evidence graph into Neo4j and
explore *everything* with Cypher filters — no aggressive caps, filtering does the
legibility work.

Generated by `translational-evidence/exports/build_neo4j_export.py` from
`data/exports/graph/{nodes,edges}.jsonl`. Regenerate with:

```bash
python3 translational-evidence/exports/build_neo4j_export.py
```

## Files

| file          | what it is |
|---------------|------------|
| `nodes.csv`   | 15,286 nodes. Base columns: `node_id,node_type,label,score,genetic_support,functional_support,translation_gap,disease_groups` (pipe-joined), `pathway_group`; then the **flat per-entity metric columns** (see below); then `provenance` (JSON string). |
| `edges.csv`   | 11,018 edges. Columns: `edge_id,source_id,target_id,edge_type,score,evidence,method,confidence`. `method` / `confidence` carry the Track A↔B bridge join metadata on `topic_*` edges (blank/Cypher-null on Track-B-internal edges). |
| `load.cypher` | `LOAD CSV WITH HEADERS` loader. Constraint + generic `:Entity` MERGE (now `SET`ting the metric columns with `toInteger/toFloat/toBoolean`) + per-type labels (`:Gene/:Variant/:Pathway/:Drug/:Trial/:Disease/:Topic`) + typed relationships + commented **example metric queries**. APOC **not** required. |

### Flat per-entity metric columns

Hoisted from `data/processed/translational-evidence/entity_metrics.jsonl` (a
transparent METRICS layer) onto the matching gene/variant/pathway nodes. They are
a **union** across types; a column is blank (Cypher `null`) for node types that
don't carry that metric. Recency metrics use `CURRENT_YEAR = 2026` (overridable
via `TE_CURRENT_YEAR`). No verdicts are baked in — compose your own from these.

| column | type | on | meaning |
|--------|------|----|---------|
| `stopped_ratio` | float | gene, pathway | stopped/terminated share of mechanism trials |
| `direction_agreement` | float | gene, variant | GWAS effect-direction agreement (1 = all agree) |
| `n_conflicting` | int | gene | min(risk, protective) direction votes |
| `n_trials` | int | gene, pathway | mechanism trial count |
| `n_drugs` | int | pathway | distinct drug/biological interventions |
| `has_approval` | bool | gene, pathway | any `APPROVED_FOR_MARKETING` trial in mechanism |
| `first_gwas_year` / `latest_gwas_year` | int | gene | gene's GWAS publication-year span |
| `n_recent_gwas` | int | gene | GWAS associations with year ≥ `CURRENT_YEAR-3` |
| `first_trial_year` / `latest_trial_year` | int | pathway | pathway's trial start-year span |
| `n_recent_trials` | int | pathway | trials starting year ≥ `CURRENT_YEAR-3` |
| `first_year` / `latest_year` | int | variant | variant's GWAS publication-year span |
| `n_recent` | int | variant | associations with year ≥ `CURRENT_YEAR-3` |
| `n_associations` / `n_studies` | int | variant | GWAS rows / distinct studies for this rsid |
| `metrics_translation_gap` | float | gene, pathway | `composite.translation_gap` (renamed to avoid clashing with the existing `translation_gap` score column, which is only populated for pathways) |

The **full** metrics object (all dotted `<group>.<name>` keys with their
`source` notes) is also attached to each node as `node['metrics']` in
`nodes.jsonl`; only the compact flat subset above is projected into `nodes.csv`.

Node types: `variant` 5786, `trial` 6841, `drug` 2111, `gene` 523, `topic` 17,
`pathway` 9, `disease` 7.
Edge types: `trial_pathway` 5271, `trial_drug` 3350, `variant_gene` 1274,
`gene_disease` 535, `topic_gene` 316, `drug_pathway` 174, `gene_pathway` 36,
`topic_disease` 33, `topic_pathway` 29.

Every node carries `provenance` (a JSON string) and scores; every edge carries a
`score` and an `evidence` label — so any result you filter to stays explainable.
The Track A↔B bridge relationships (`TOPIC_GENE` / `TOPIC_PATHWAY` /
`TOPIC_DISEASE`) additionally carry `r.method` + `r.confidence`, so you can keep
only structured, high-confidence links, e.g.
`MATCH (t:Topic)-[r]->(x) WHERE r.confidence = 'high' AND r.method <> 'regex_symbol_match' RETURN ...`.
The `TOPIC_DISEASE` relationship (topic → disease group, via the structured
`mesh_ui_join`) is new. See `translational-evidence/exports/LINK_METHODS.md`.

## 1. Start Neo4j with this directory mounted as the import folder

Run from the **repo root** (so `$PWD` resolves correctly):

```bash
docker run -it --rm -p7474:7474 -p7687:7687 \
  -e NEO4J_AUTH=neo4j/testpassword \
  -v "$PWD/data/exports/graph/neo4j:/var/lib/neo4j/import" \
  neo4j:5
```

`nodes.csv` / `edges.csv` are now visible to Cypher as `file:///nodes.csv` and
`file:///edges.csv` (Neo4j resolves `file:///` against the import directory).

## 2. Load the graph

Either pipe the script into `cypher-shell` (from the repo root, in a second
terminal):

```bash
cat data/exports/graph/neo4j/load.cypher | \
  docker exec -i $(docker ps -q --filter ancestor=neo4j:5) \
  cypher-shell -u neo4j -p testpassword
```

…or, if you have a local `cypher-shell` on your PATH:

```bash
cat data/exports/graph/neo4j/load.cypher | cypher-shell -u neo4j -p testpassword
```

…or open **Neo4j Browser** at <http://localhost:7474> (login
`neo4j` / `testpassword`) and paste the contents of `load.cypher`.

The script prints two sanity tables at the end (node counts by label, relationship
counts by type). Expect 15,294 nodes and 11,018 relationships.

> Load is one transaction per pass; the whole graph is small (~10.7k edges) so a
> default `neo4j:5` container handles it fine. For much larger exports you would
> wrap the relationship `CREATE`s in `CALL { ... } IN TRANSACTIONS`.

## 3. Explore EVERYTHING with filters

Every query below is a *filter over the full graph* — tighten or loosen the
thresholds to see more or less. All scores are in `[0,1]` unless noted.

### a. Under-translated pathways (high translation gap, few or no drugs)
Pathways where the biology is supported but little/nothing has reached the clinic.

```cypher
MATCH (p:Pathway)
OPTIONAL MATCH (:Drug)-[dp:DRUG_PATHWAY]->(p)
WITH p, count(dp) AS drug_edges
WHERE p.translation_gap >= 0.05
RETURN p.label, p.translation_gap, p.pathway_group, drug_edges
ORDER BY p.translation_gap DESC, drug_edges ASC;
```

### b. Microglia genes with strong genetic support
Filter genes by pathway group and a genetic-support threshold.

```cypher
MATCH (g:Gene)
WHERE g.pathway_group CONTAINS 'microglia'
  AND g.genetic_support > 0.5
RETURN g.label, g.genetic_support, g.functional_support, g.disease_groups
ORDER BY g.genetic_support DESC;
```

### c. All evidence for a gene: the full variant -> gene -> pathway -> drug chain
Swap `'APOE'` for any gene symbol. Shows how genetics connects through to drugs.

```cypher
MATCH path = (v:Variant)-[:VARIANT_GENE]->(g:Gene {label:'APOE'})
             -[:GENE_PATHWAY]->(p:Pathway)<-[:DRUG_PATHWAY]-(d:Drug)
RETURN v.label AS variant, g.label AS gene, p.label AS pathway,
       d.label AS drug
LIMIT 200;
```

### d. Genes shared across disease groups
Pleiotropic genes wired to more than one dementia disease group.

```cypher
MATCH (g:Gene)-[:GENE_DISEASE]->(dis:Disease)
WITH g, collect(DISTINCT dis.label) AS diseases
WHERE size(diseases) > 1
RETURN g.label, g.genetic_support, diseases
ORDER BY size(diseases) DESC, g.genetic_support DESC;
```

### e. Trials by phase for a mechanism (pathway) group
All trials mapped to the `amyloid` pathway, broken down by phase. `phases` lives
inside the trial `provenance` JSON string, so filter/inspect it there.

```cypher
MATCH (t:Trial)-[:TRIAL_PATHWAY]->(p:Pathway {pathway_group:'amyloid'})
RETURN t.label AS trial, t.provenance AS provenance_json
ORDER BY trial
LIMIT 200;
```

To slice by a specific phase, filter on the JSON text (APOC-free):

```cypher
MATCH (t:Trial)-[:TRIAL_PATHWAY]->(p:Pathway {pathway_group:'amyloid'})
WHERE t.provenance CONTAINS '"PHASE3"'
RETURN t.label, t.provenance;
```

### f. The full evidence chain for APOE (everything within 2 hops)
The complete neighbourhood — variants, pathways, diseases, topics, and (via
pathways) drugs and trials.

```cypher
MATCH (g:Gene {label:'APOE'})
MATCH path = (g)-[*1..2]-(neighbour)
RETURN path
LIMIT 500;
```

### g. Drugs active in an under-translated pathway
Combine a pathway filter with its drug and trial evidence to spot where effort
is (or is not) going.

```cypher
MATCH (p:Pathway)<-[:DRUG_PATHWAY]-(d:Drug)
WHERE p.translation_gap >= 0.05
OPTIONAL MATCH (t:Trial)-[:TRIAL_DRUG]->(d)
RETURN p.label AS pathway, d.label AS drug, count(t) AS trials
ORDER BY pathway, trials DESC;
```

### h. Highest-confidence GWAS variant->gene links for Alzheimer
Rank the raw genetic edges by score, filtered to a disease group.

```cypher
MATCH (v:Variant)-[r:VARIANT_GENE]->(g:Gene)
WHERE 'alzheimer' IN g.disease_groups
RETURN v.label AS variant, g.label AS gene, r.score AS score, r.evidence
ORDER BY score DESC
LIMIT 50;
```

## 4. Query the per-entity METRICS

The flat metric columns are first-class node properties, so you can filter and
rank on them directly. These are **transparent signals**, not verdicts — read
them side by side and draw your own conclusions. (The same queries ship commented
out at the bottom of `load.cypher`.)

### i. Genetically supported but clinically stalled genes
Strong genetic support yet a high share of stopped/terminated trials in the gene's
mechanism.

```cypher
MATCH (g:Gene)
WHERE g.genetic_support >= 0.7 AND g.stopped_ratio >= 0.3 AND g.n_trials >= 5
RETURN g.label, g.genetic_support, g.stopped_ratio, g.n_trials,
       g.metrics_translation_gap
ORDER BY g.stopped_ratio DESC, g.genetic_support DESC
LIMIT 25;
```

### j. Recently-emerging loci
Variants whose latest GWAS is in the last ~3 years (`latest_year >= CURRENT_YEAR-2`,
i.e. 2024) with at least one recent association.

```cypher
MATCH (v:Variant)
WHERE v.latest_year >= 2024 AND v.n_recent >= 1
RETURN v.label, v.latest_year, v.n_recent, v.n_associations, v.n_studies
ORDER BY v.latest_year DESC, v.n_associations DESC
LIMIT 25;
```

### k. Direction-conflict genes
GWAS effect directions disagree across studies (low `direction_agreement` with
real conflicting evidence).

```cypher
MATCH (g:Gene)
WHERE g.direction_agreement IS NOT NULL
  AND g.direction_agreement < 0.7 AND g.n_conflicting >= 2
RETURN g.label, g.direction_agreement, g.n_conflicting, g.genetic_support
ORDER BY g.n_conflicting DESC, g.direction_agreement ASC
LIMIT 25;
```

## Notes / caveats

- `provenance` is stored as a **string** (the original JSON). Neo4j core cannot
  parse JSON without APOC, so filter it with `CONTAINS` or read it back in your
  app. Numeric scores are already first-class node/edge properties.
- `disease_groups` is stored as a Neo4j **list** (split from the pipe-joined CSV
  column), so `'alzheimer' IN n.disease_groups` works directly.
- Relationships are loaded with `CREATE` (not `MERGE`): 5 `topic_gene` edges in
  the source share a topic+gene pair but carry distinct evidence
  (`gene_mention` vs `paper_overlap`). Their `edge_id`s were made unique with a
  `#2` suffix so no evidence is lost.
